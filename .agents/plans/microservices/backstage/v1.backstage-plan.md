---

# PURPOSE OF BACKSTAGE

Backstage provides visibility into the ingestion pipeline.

It listens to platform events and records the lifecycle of each track.

Responsibilities:

```
1. Subscribe to platform events
2. Record ingestion lifecycle
3. Store pipeline history
4. Expose internal monitoring APIs
```

Backstage **does NOT**:

```
process audio
perform AI inference
orchestrate services
store media files
emit ingestion commands
```

Backstage is a **read-model microservice**.

# DATABASE

Backstage uses MongoDB.

MongoDB stores the ingestion read model.

Primary collection:

```
track_pipelines
```

Each document represents a track and its pipeline history.

Example structure:

```json
{
  "trackId": "uuid",
  "status": "processing",
  "currentStage": "transcoding",
  "events": [
    {
      "eventType": "track.uploaded",
      "timestamp": "2026-03-13T12:00:00Z",
      "payload": {}
    }
  ],
  "createdAt": "timestamp",
  "updatedAt": "timestamp"
}
```

# INTERFACE LAYER

Contains entry points into the microservice.

Two interfaces exist.

**Event Handlers**

Handlers consume events from NATS.

Examples:

```
TrackUploadedHandler
TrackTranscodingCompletedHandler
TrackReadyHandler
```

Handlers translate external events into application use cases.

**HTTP API**

Backstage exposes monitoring endpoints.

Examples:

```
GET /pipelines/:trackId
GET /pipelines
GET /pipelines/active
GET /pipelines/failed
```

These endpoints allow internal tools to inspect the ingestion pipeline.

# EVENT STORAGE STRATEGY

Backstage stores all events associated with a track.

Events are appended to the pipeline history.

Benefits:

```
full audit trail
pipeline debugging
lifecycle reconstruction
```

---

# ERROR HANDLING

Event processing must be resilient.

If an event cannot be processed:

```
1. Log the error
2. Skip the event
3. Continue consuming events
```

The service must **never crash** due to a malformed event.
